# Prompt: Build InariWatch E-Commerce Tester App

You are building a demo e-commerce app (mini sticker shop) designed to showcase every capability of InariWatch, an AI-powered error monitoring and auto-remediation platform. The app has two sides: a functional store that works correctly, and a chaos panel that activates REAL bugs which InariWatch detects, diagnoses, and fixes automatically.

## CRITICAL CONTEXT

This app exists in a **separate GitHub repo** (NOT inside the InariWatch monorepo). InariWatch needs GitHub access to read code and create PRs. The app must be deployed on Vercel with `@inariwatch/capture` installed.

## 5. CHAOS PANEL (`/admin/chaos`)

A dashboard with toggle switches. Each toggle activates a REAL bug in the codebase. The bugs are NOT `throw new Error("fake")` — they are actual code defects that InariWatch's AI can diagnose and fix.

### HOW TOGGLES WORK

Each bug has two implementations in the code: the correct one and the buggy one. A runtime check reads the toggle state from DB:

```typescript
// lib/chaos/toggles.ts
import { db, chaosToggles } from "@/lib/db"
import { eq } from "drizzle-orm"

const cache = new Map<string, { active: boolean; ts: number }>()

export async function isChaosActive(bugId: string): Promise<boolean> {
  const cached = cache.get(bugId)
  if (cached && Date.now() - cached.ts < 5000) return cached.active
  
  const [toggle] = await db
    .select()
    .from(chaosToggles)
    .where(eq(chaosToggles.bugId, bugId))
    .limit(1)
  
  const active = toggle?.isActive ?? false
  cache.set(bugId, { active, ts: Date.now() })
  return active
}
```

### IMPORTANT: How InariWatch fixes these bugs

When InariWatch's AI reads the code and sees the `isChaosActive()` check, it will understand the conditional logic. The AI will:
1. Read the stack trace pointing to the buggy code path
2. See that when `isChaosActive("sql-injection")` is true, the code uses string concatenation
3. Generate a fix that replaces the buggy path with parameterized queries
4. The fix keeps the toggle but fixes the vulnerability in the "active" branch

This is realistic because in real codebases, feature flags often guard experimental code that has bugs — InariWatch fixes the bug inside the flag, not the flag itself.

### BUG 1: SQL Injection in Search (Security Scan)
**ID:** `sql-injection`
**Severity:** Critical
**InariWatch path:** Security scan (regex pattern #5: `SELECT.*\+.*FROM`) + AI security review (category 1: SQL injection)
**Trigger:** User searches for `'; DROP TABLE products--`

**Code in `api/search/route.ts`:**
```typescript
export async function GET(req: Request) {
  const { searchParams } = new URL(req.url)
  const q = searchParams.get("q") || ""
  
  if (await isChaosActive("sql-injection")) {
    // BUG: String concatenation in SQL query
    const results = await db.execute(
      sql.raw(`SELECT * FROM products WHERE name ILIKE '%${q}%' OR description ILIKE '%${q}%'`)
    )
    return NextResponse.json(results.rows)
  }
  
  // CORRECT: Parameterized query
  const results = await db
    .select()
    .from(products)
    .where(or(
      ilike(products.name, `%${q}%`),
      ilike(products.description, `%${q}%`)
    ))
  return NextResponse.json(results)
}
```

**What capture sends:** Error event with title "error: syntax error at or near...", body with full stack trace, breadcrumbs showing the search fetch, request context with the malicious query.

**What InariWatch does:**
1. Auto-analyze: "SQL injection vulnerability — user input concatenated directly into query string"
2. Security scan: Regex pattern detects `SELECT.*+.*FROM`, AI review flags SQL injection
3. Remediation: Replaces string concatenation with Drizzle ORM parameterized query
4. Self-review: High confidence (pattern is textbook SQL injection)

### BUG 5: Race Condition in Stock (Logic Bug)
**ID:** `race-stock`
**Severity:** High
**InariWatch path:** Auto-analyze + remediation (medium-high confidence: race condition pattern)
**Trigger:** Two users buy the last item simultaneously (simulate with rapid double-click)

**Code in `api/checkout/route.ts`:**
```typescript
async function processOrder(userId: string, items: CartItem[]) {
  if (await isChaosActive("race-stock")) {
    // BUG: Check-then-act without transaction — race condition
    for (const item of items) {
      const [product] = await db.select().from(products).where(eq(products.id, item.productId))
      if (product.stock < item.quantity) {
        throw new Error(`Insufficient stock for ${product.name}`)
      }
    }
    // Another request can buy between check and update
    for (const item of items) {
      await db.update(products)
        .set({ stock: sql`stock - ${item.quantity}` })
        .where(eq(products.id, item.productId))
    }
    return
  }
  
  // CORRECT: Atomic transaction with SELECT FOR UPDATE
  await db.transaction(async (tx) => {
    for (const item of items) {
      const [product] = await tx.select().from(products)
        .where(eq(products.id, item.productId))
        .for("update")
      if (product.stock < item.quantity) {
        throw new Error(`Insufficient stock for ${product.name}`)
      }
      await tx.update(products)
        .set({ stock: product.stock - item.quantity })
        .where(eq(products.id, item.productId))
    }
  })
}
```

**What InariWatch does:**
1. Auto-analyze: "Race condition — stock check and update are not atomic. Concurrent requests can oversell"
2. Remediation: Wraps in transaction with `FOR UPDATE` lock
3. Medium-high confidence: Requires understanding async flow, but pattern is recognizable

### BUG 7: Unhandled Promise Rejection (Runtime Error)
**ID:** `unhandled-promise`
**Severity:** Critical
**InariWatch path:** Auto-analyze + remediation (high confidence: missing error handling)
**Trigger:** Product page loads when DB is temporarily slow/unavailable

**Code in `(store)/products/[id]/page.tsx`:**
```typescript
async function ProductPage({ params }: { params: { id: string } }) {
  if (await isChaosActive("unhandled-promise")) {
    // BUG: No error handling on DB query — unhandled rejection crashes page
    const product = await db.select().from(products).where(eq(products.id, params.id))
    const reviews = await db.select().from(reviews).where(eq(reviews.productId, params.id))
    // If DB is slow, this throws unhandled
    const relatedProducts = await fetch(`https://api.example.com/related/${params.id}`).then(r => r.json())
    return <ProductDetail product={product[0]} reviews={reviews} related={relatedProducts} />
  }
  
  // CORRECT: Graceful error handling
  try {
    const [product] = await db.select().from(products).where(eq(products.id, params.id))
    if (!product) return notFound()
    const reviews = await db.select().from(reviews).where(eq(reviews.productId, params.id))
    let relatedProducts = []
    try {
      relatedProducts = await fetch(`https://api.example.com/related/${params.id}`).then(r => r.json())
    } catch { /* external API failure is non-critical */ }
    return <ProductDetail product={product} reviews={reviews} related={relatedProducts} />
  } catch {
    return <ErrorFallback message="Could not load product" />
  }
}
```

**What InariWatch does:**
1. Auto-analyze: "Unhandled promise rejection — external API call has no error handling"
2. Fix: Wrap in try-catch, add fallback for non-critical fetch
3. High confidence: Stack trace shows exact rejection point

### BUG 8: Prototype Pollution (Security Scan)
**ID:** `prototype-pollution`
**Severity:** High
**InariWatch path:** Security scan (regex pattern #8: `__proto__` + pattern #9: `constructor[`) + AI review (category 8: prototype pollution)
**Trigger:** POST to `/api/cart` with `{"__proto__": {"isAdmin": true}}`

**Code in `api/cart/route.ts`:**
```typescript
export async function POST(req: Request) {
  const body = await req.json()
  
  if (await isChaosActive("prototype-pollution")) {
    // BUG: Deep merge without sanitization allows __proto__ injection
    const defaults = { quantity: 1, notes: "" }
    const merged = deepMerge(defaults, body)
    // body with __proto__ pollutes Object.prototype
    return NextResponse.json(merged)
  }
  
  // CORRECT: Only pick known fields
  const { productId, quantity = 1 } = body
  if (!productId) return NextResponse.json({ error: "productId required" }, { status: 400 })
  // ... add to cart
}

function deepMerge(target: any, source: any): any {
  for (const key of Object.keys(source)) {
    if (typeof source[key] === "object" && source[key] !== null) {
      target[key] = deepMerge(target[key] || {}, source[key])
    } else {
      target[key] = source[key]
    }
  }
  return target
}
```

**What InariWatch does:**
1. Security scan: Regex detects `__proto__` pattern, AI flags prototype pollution
2. Fix: Replace deepMerge with explicit field destructuring
3. High confidence: Pattern is well-known, fix is straightforward

### What the AI sees when reading the code
The AI reads the full file content from GitHub. It will see:

If you want the AI to generate a "cleaner" fix (removing the toggle entirely), you can structure the code without the conditional — just have the buggy version as the only code. But then you lose the ability to toggle bugs on/off without redeploying.

### Capture event schema (what gets sent to InariWatch)
Each error event includes:
- `fingerprint`: SHA-256 hash of normalized error (strips UUIDs, timestamps, paths, line numbers)
- `title`: "ErrorName: message"
- `body`: Full stack trace
- `severity`: "critical" | "warning" | "info"
- `breadcrumbs`: Last 30 actions (console logs, fetch calls) with timestamps
- `git`: { commit, branch, message, dirty } (injected at build time by withInariWatch)
- `env`: { node version, platform, arch, CPU count, memory, heap, uptime }
- `user`: { id, role } (from scope, email stripped for PII)
- `tags`: { page, custom tags }
- `request`: { method, url, headers (redacted), body (redacted) }
- `context`: Arbitrary custom data passed to captureException

### InariWatch remediation capabilities (verified from source)
The AI can fix with HIGH confidence (80%+):
- Null reference errors (add optional chaining, null checks)
- SQL injection (replace concatenation with parameterized)
- XSS (replace dangerouslySetInnerHTML with text rendering)
- Hardcoded secrets (replace with env var)
- Missing auth checks (add session validation)
- Off-by-one errors (fix arithmetic)
- Missing error handling (add try-catch)
- Wrong regex flags, type coercions, missing returns

The AI fixes with MEDIUM confidence (40-70%):
- Race conditions (add transactions — recognizable pattern but needs DB understanding)
- Complex state management, API contract mismatches

The AI CANNOT reliably fix:
- Distributed system bugs, complex auth flows, crypto issues, performance regressions, business logic errors
